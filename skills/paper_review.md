# Skill: 论文审稿与迭代（paper_review）

## 目的
以 IEEE RAL 审稿人视角审查论文草稿，识别问题，输出结构化的修改建议，支持多轮迭代改进。
调用时机：每完成一节草稿、完成初稿全文、收到审稿意见后修订时。

## 审稿维度与检查清单

### 维度 1：技术正确性（Technical Correctness）

#### 1.1 公式推导
- [ ] iLQR 后向传递的 Riccati 递推公式正确
- [ ] Tube 三项代价的 Gauss-Newton 导数推导无误
- [ ] 动力学线性化 A_k, B_k 的定义与代码实现一致
- [ ] 状态空间定义 x = [q, q̇] 与公式中的维度匹配
- [ ] 终端代价 `(1 - tube_ratio) * l_N(x)` 的含义说清

#### 1.2 算法描述
- [ ] MPC 重规划频率与 replan_interval 的关系说清
- [ ] 异步重规划的数据竞争问题是否讨论（独立 MjData）
- [ ] 安全滤波器的 β 衰减策略是否有理论保证（或说明是工程启发式）
- [ ] Tube 窗口搜索的可达性过滤（IK 裕度 ≥ 3°）是否在论文中说明

#### 1.3 数值合理性
- [ ] 命中率数据：100% 是否有统计意义（20 个 seed 足够？）
- [ ] 位置误差数据：均值 < 5cm 是否在球拍半径（120mm）内？
- [ ] 实时性数据：50-80ms 重规划时间在不同硬件上的可复现性
- [ ] max_qdot_ratio 0.92-1.00× 是否真实未超限

---

### 维度 2：创新性与贡献（Novelty and Contribution）

#### 2.1 贡献清晰度
- [ ] C1（空间走廊 Tube）：与已有 tube MPC 的本质区别是否在 1-2 句话说清？
- [ ] C2（安全滤波）：与 CBF（控制屏障函数）的区别是否明确？
- [ ] C3（实时 MPC）：异步重规划是否是新贡献？（很多 MPC 系统都有）
- [ ] C4（实验验证）：纯实验贡献可能不够，需与 C1-C3 绑定

#### 2.2 基线对比充分性
- [ ] 是否与纯 PD 跟踪方法对比？（最低要求）
- [ ] 是否与标准 iLQR（无 Tube）对比？（已有 exp6 消融可支持）
- [ ] 是否与 RL 方法对比？（可选，但审稿人可能问）
- [ ] 是否与 Jacobian 转矩（JT）方法对比？（代码中已用 JT 作为后备）

#### 2.3 创新性风险点
- [ ] 风险1：审稿人认为"空间走廊"只是加权多目标优化，不是真正的 tube MPC
  - 应对：强调不确定性管道 σ(t) 的显式建模和 hinge loss 的 corridor 设计
- [ ] 风险2：审稿人认为多层安全滤波是工程堆叠，缺乏理论保证
  - 应对：补充安全性证明或引用安全滤波相关文献
- [ ] 风险3：审稿人认为异步重规划不是新贡献
  - 应对：弱化 C3 的创新性表述，改为"实现细节"而非"贡献"

---

### 维度 3：实验充分性（Experimental Rigor）

#### 3.1 统计显著性
- [ ] 每组实验 ≥ 20 seeds（是否足够？考虑增加到 50）
- [ ] 报告均值 ± 标准误（不是标准差）
- [ ] 命中率 95% CI（置信区间）是否重叠（不同条件间）

#### 3.2 实验设置公平性
- [ ] 所有方法使用相同的随机种子序列
- [ ] 球拍初始位姿一致
- [ ] Tube on/off 的其他参数完全相同（仅 tube 开关不同）
- [ ] 球速定义一致（"球到达击打点时的水平速度"）

#### 3.3 可复现性
- [ ] 配置参数完整公开（附 YAML 文件或参数表）
- [ ] 随机种子公开
- [ ] MuJoCo 模型文件公开
- [ ] 代码仓库链接（或补充材料）

#### 3.4 实验覆盖度
- [ ] 是否覆盖了不同球速（慢/中/快）？
- [ ] 是否覆盖了不同发球位置（serve_box 随机）？
- [ ] 是否测试了极限情况（极高/低球速、大扰动）？
- [ ] 是否有失败案例分析（不只是报告成功率）？

---

### 维度 4：写作质量（Writing Quality）

#### 4.1 逻辑连贯性
- [ ] Introduction 的动机链完整：背景 → 问题 → 局限 → 方法 → 贡献
- [ ] Method 节的叙述顺序：总体框架 → 子组件 → 实现细节
- [ ] Experiments 节的结果能支撑 Claims
- [ ] Conclusion 准确反映实验结果（不过度声称）

#### 4.2 图表质量
- [ ] 所有图表在灰度打印下可区分（色盲友好已处理？）
- [ ] 图表标题和标注字体大小 ≥ 7pt
- [ ] 每个图表在正文中被引用
- [ ] 表格数据与正文描述一致

#### 4.3 英文质量
- [ ] 无语法错误（时态一致、主谓一致）
- [ ] 无中式英语（避免 "has a good performance" 等）
- [ ] 术语全文一致（不混用 "tube corridor" 和 "spatial envelope"）
- [ ] 被动语态和主动语态的使用恰当

#### 4.4 格式合规
- [ ] 页数 ≤ 8 页（不含参考文献）
- [ ] 双栏格式正确
- [ ] 参考文献格式 IEEEtran
- [ ] 图表位置合理（不跨栏截断）

---

### 维度 5：与相关工作的差异化（Differentiation）

#### 5.1 必须对比的已有工作
- [ ] Todorov & Li (2005) iLQG — 本文的基础求解器
- [ ] Mayne et al. (2011) Tube-based robust MPC — 本文 tube 机制的灵感来源
- [ ] Tassa et al. (2012) Synthesis and stabilization — iLQR 在机器人中的应用
- [ ] 网球/乒乓球机器人的代表性工作（至少 2-3 篇）

#### 5.2 需明确说明的差异
- [ ] 本文 tube vs 传统 tube MPC：空间走廊 vs 时间锁定
- [ ] 本文安全滤波 vs CBF：逐步衰减 vs 优化约束
- [ ] 本文 iLQR vs DDP：无二阶项 vs 有二阶项

---

### 维度 6：实际部署价值（Practical Impact）

#### 6.1 Sim-to-Real 讨论
- [ ] 仿真与真实机器人的差距是否讨论？
  - MuJoCo 刚体动力学 vs 真实柔性关节
  - 球轨迹预测模型（抛物线 vs 含空气阻力）
  - 传感器延迟和噪声
- [ ] 是否有 Sim-to-Real 迁移计划？
  - 域随机化？
  - 系统辨识？

#### 6.2 安全性讨论
- [ ] 安全滤波器的完备性：有无遗漏的危险情况？
- [ ] TCP 速度限制 1.8 m/s 的来源是否说明？
- [ ] 紧急停止机制是否描述？

#### 6.3 泛化性
- [ ] 方法是否适用于其他机器人（UR5e、Franka 等）？
- [ ] 方法是否适用于其他球速范围？
- [ ] 方法是否适用于其他动态操作任务？

## 预设审稿人质疑与应对

以下是 IEEE RAL 审稿人可能提出的关键质疑，以及建议的应对策略：

### 质疑 1："空间走廊只是加权代价函数，不是真正的 tube MPC"
- **应对**：在 Method 节明确区分：(1) 传统 tube MPC 是约束满足问题，本文是代价正则化方法；(2) 我们的不确定性管道 σ(t) 提供了显式的鲁棒性保证；(3) 实验表明 ±50ms 扰动下命中率从 50% 提升到 85%，这是加权代价无法单独实现的
- **修改位置**：Section IV-B 末段、Discussion

### 质疑 2："缺少与 RL 或其他学习方法的对比"
- **应对策略A**（推荐）：在 Related Work 中讨论 RL 方法，在 Discussion 中说明本文聚焦传统最优控制的优势（可加入硬约束、无需训练、可解释性）
- **应对策略B**：补充简单的 RL/PD 基线实验（如有时间）

### 质疑 3："仿真结果能否迁移到真实机器人？"
- **应对**：Discussion 节专门讨论 Sim-to-Real gap：(1) 安全滤波器确保真实部署安全；(2) MuJoCo 物理精度较高；(3) 未来工作包括真实 RM-65B 实验

### 质疑 4："200Hz 控制频率下的计算是否真的实时？"
- **应对**：Fig.7 数据清楚表明所有稳态重规划在预算内；真实机器人去掉 MuJoCo 仿真开销后每步仅需 ~2ms

### 质疑 5："球轨迹预测过于简单（抛物线、无空气阻力）"
- **应对**：这正是 Tube 机制的价值所在——简单的预测模型 + Tube 鲁棒性补偿预测误差；实验4的扰动测试验证了这一点

### 质疑 6："固定球速和固定场景，泛化性如何？"
- **应对**：serve_box 模式已覆盖 8m×0.2m×0.3m 的随机发球区域；多球速测试覆盖 7-30 m/s；多 seed 统计保证泛化性

## 修改优先级定义

| 优先级 | 含义 | 修改时限 |
|--------|------|---------|
| P0 | 阻塞性问题，不修改将导致退稿 | 立即修改 |
| P1 | 重要问题，影响论文质量 | 当轮修改 |
| P2 | 改进建议，提升论文质量 | 下轮修改 |
| P3 | 锦上添花，可选修改 | 有空再改 |

## 审稿记录管理

所有审稿报告保存在 `paper/reviews/` 目录：

```
paper/reviews/
├── review_v1_technical.md    # 第一轮：技术审稿
├── review_v2_experiment.md   # 第二轮：实验审稿
├── review_v3_writing.md      # 第三轮：写作审稿
├── review_v4_format.md       # 第四轮：格式审稿
└── review_v5_final.md        # 最终检查
```

每次审稿后：
1. 保存审稿报告到对应文件
2. 更新 `paper/notes.md` 中的修改追踪表
3. 在代码中修改对应问题后，标注 `[Fixed: review_vN M1]`
